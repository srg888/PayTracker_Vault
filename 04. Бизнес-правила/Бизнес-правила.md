# Бизнес-правила

## Назначение

Этот документ содержит бизнес-правила PayTracker. Каждое правило имеет уникальный идентификатор. Изменение бизнес-правила должно отражаться в истории изменений документа и, если необходимо, в соответствующем [[Архитектурные решения|архитектурном решении]].

# 1. Пользователи и роли
См. [[Роли и права]], [[Матрица прав]].

## BR-001. Авторизация пользователя
Пользователь должен быть идентифицирован до выполнения действий, требующих авторизации.

## BR-002. Активный пользователь
Неактивный пользователь не может: создавать новые заявки; быть назначенным исполнителем; выполнять действия по заявке; создавать делегирование.

## BR-003. Исполнитель
Исполнителем может быть только активный пользователь с ролью, разрешённой Матрицей прав. Перед [[03. Назначение исполнителя|назначением исполнителя]] сервер обязан проверить его роль и активность.

# 2. Заявки
См. [[00. Создание заявки|Создание заявки]].

## BR-010. Создание заявки
Заявка создаётся в статусе `Черновик`.

## BR-011. Редактирование черновика
До подачи заявки Автор может изменять её поля.

## BR-012. Неизменяемость поданной заявки
После перехода заявки из `Черновик` в `Новая заявка` поля, влияющие на смысл операции, не могут быть изменены обычным редактированием. К таким полям относятся: тип заявки; сумма; валюта; получатель; способ платежа; платёжный агент; назначение платежа; срок исполнения. Если требуется изменить существенные параметры поданной заявки, должна быть предусмотрена отдельная процедура.

## BR-013. Номер заявки
Номер заявки должен быть уникальным. Номер не должен формироваться на основании количества существующих записей. Удаление заявки не должно приводить к повторному использованию её номера (см. [[Архитектурные решения|ADR-005]]).

## BR-014. Автор заявки
В текущей версии системы Автор заявки одновременно является Заказчиком (см. [[Словарь предметной области]]).

# 3. Исполнители
См. [[03. Назначение исполнителя|Назначение исполнителя]].

## BR-020. Первичное назначение
Исполнитель назначается Руководителем или пользователем, имеющим действующие делегированные полномочия Руководителя.

## BR-021. Переназначение
Исполнитель может быть изменён после первоначального назначения. Переназначение должно быть разрешено только уполномоченному пользователю, фиксироваться в аудите и сохранять сведения о предыдущем исполнителе.

## BR-022. История исполнителей
История назначений должна позволять определить, кто был назначен, кем, когда, до какого момента и кто был назначен после него.

# 4. Делегирование
См. [[04. Делегирование полномочий Руководителя|Делегирование полномочий Руководителя]].

## BR-030. Одно активное делегирование
Одновременно может существовать не более одного активного делегирования полномочий Руководителя.

## BR-031. Период делегирования
Делегирование действует в период: `дата начала <= текущая дата/время <= дата окончания`.

## BR-032. Отзыв делегирования
Делегирование может быть отозвано досрочно Руководителем. Дата и факт отзыва фиксируются в аудите.

## BR-033. Делегированные полномочия
Делегированный пользователь получает только те полномочия, которые прямо предусмотрены Матрицей прав.

# 5. Статусы
См. [[Статусы заявки]].

## BR-040. Допустимые переходы
Переходы между статусами разрешены только согласно определённой статусной машине. Нельзя установить статус напрямую, минуя разрешённый переход.

## BR-041. История статусов
Каждый переход между статусами должен сохраняться. Для перехода фиксируются: предыдущий статус; новый статус; пользователь; дата и время; причина или комментарий, если они обязательны.

# 6. Документы
См. [[Справочник типов документов]], [[09. Проверка комплектности и закрытие заявки|Проверка комплектности и закрытие заявки]].

## BR-050. Обязательные документы
Заявка не может быть закрыта, если отсутствует обязательный документ.

## BR-051. Серверная проверка документов
Все правила обязательности документов должны проверяться на сервере. Нельзя полагаться только на ограничения пользовательского интерфейса.

## BR-052. Индивидуальные требования
Для конкретной заявки может быть создано индивидуальное правило обязательности документа. Такое правило должно содержать заявку, тип документа, значение обязательности, автора изменения, дату и причину изменения.

## BR-053. Загрузка документов
При загрузке документа сервер обязан проверить права пользователя, статус заявки, допустимость типа документа, размер файла, допустимый формат, ограничения по количеству файлов и принадлежность документа к заявке.

# 7. Денежные значения
См. [[01. Поля заявки — Платёж|Поля заявки — Платёж]], [[08. Фактическое исполнение платежа|Фактическое исполнение платежа]].

## BR-060. Денежные суммы
Денежные значения должны обрабатываться с использованием точного десятичного типа. Использование бинарного `float` для финансовых расчётов запрещено (см. [[Архитектурные решения|ADR-004]]).

## BR-061. Курс валюты
Для каждого расчёта должна быть сохранена информация об использованном курсе, источнике курса, дате курса и признаке актуальности.

## BR-062. Фактическая сумма в рублях
Если сумма операции пересчитывается в рубли, итоговая сумма в рублях должна быть сохранена как отдельное значение. Она не должна каждый раз пересчитываться заново по текущему курсу.

# 8. Аудит
См. [[Аудит]].

## BR-070. Неизменяемость аудита
Записи аудита не должны редактироваться обычными пользователями.

## BR-071. Аудируемые действия
Минимальный набор аудируемых действий: создание заявки; подача заявки; изменение заявки; назначение исполнителя; переназначение исполнителя; изменение статуса; отклонение заявки; запрос уточнения; ответ на уточнение; подтверждение исполнения; загрузка документа; удаление документа; изменение требований к документам; создание делегирования; отзыв делегирования; вход пользователя; выход пользователя.

# 9. Фактическое исполнение
См. [[08. Фактическое исполнение платежа|Фактическое исполнение платежа]].

## BR-080. Разделение заявки и исполнения
Заявка описывает запрошенное действие. Фактическое исполнение описывает реально выполненное действие. Эти понятия не должны смешиваться.

## BR-081. Один или несколько платежей
В текущей версии системы необходимо явно определить, допускает ли одна платёжная заявка только одно фактическое исполнение или несколько частичных исполнений (см. [[Архитектурные решения|ADR-006]]). Если допускаются частичные исполнения, фактические платежи должны быть отдельными сущностями.

## BR-082. Завершение исполнения
Исполнитель может завершить исполнение только после фиксации необходимых данных о фактическом результате.

## BR-083. Закрытие заявки
Заявка может быть закрыта только после: 1) завершения исполнения; 2) подтверждения результата, если оно требуется; 3) проверки обязательных документов; 4) выполнения других обязательных условий.

# 10. Отчётность
См. [[Отчётность]].

## BR-090. Источник отчётности
Отчёты должны формироваться на основании сохранённых данных операций и истории системы.

## BR-091. Фактические суммы
Для финансовых отчётов используются фактически исполненные суммы и сохранённые значения курсов. Текущий курс валюты не должен изменять историческую отчётность.
