# Матрица прав

Детализация [[Роли и права|ролей]] по конкретным действиям, привязанным к этапам [[01. Жизненный цикл заявки|жизненного цикла заявки]].

## Обозначения
- **✓** — действие разрешено.
- **—** — действие запрещено.
- **D** — разрешено только в рамках действующего [[04. Делегирование полномочий Руководителя|делегирования]].

## Правила применения матрицы

### 1. Проверка выполняется на сервере
Скрытие кнопки в интерфейсе не является проверкой права. Каждый API endpoint должен самостоятельно проверять: наличие авторизованного пользователя, его роль, активность пользователя, действующее делегирование, статус заявки, принадлежность пользователя к заявке (см. [[Архитектурные решения|ADR-003]]).

### 2. Делегирование не меняет роль пользователя
Делегирование не изменяет значение роли пользователя в базе данных:

```text
role = Исполнитель
```

остаётся неизменным. При проверке права система дополнительно проверяет: "есть ли активное делегирование полномочий Руководителя?" (см. [[04. Делегирование полномочий Руководителя|Делегирование полномочий Руководителя]]).

### 3. Доступ к данным и право действия — разные понятия
Пользователь может иметь право просматривать заявку, но не иметь права изменять её. Доступ к объекту и возможность совершения действия проверяются отдельно.

### 4. Доступ к аудиту
Полный журнал [[Аудит|аудита]] доступен только Руководителю и пользователю с действующим делегированием его полномочий. История конкретной заявки может отображаться в карточке заявки в объёме, разрешённом [[Бизнес-правила|бизнес-правилами]].

## Приоритеты
Если одновременно применимы несколько правил доступа, действует наиболее ограничительное:

```text
Пользователь имеет роль Руководителя
        ↓
Но заявка находится в статусе,
в котором действие запрещено
        ↓
Действие запрещено
```

Роль сама по себе не отменяет ограничений [[Статусы заявки|статусной машины]].
