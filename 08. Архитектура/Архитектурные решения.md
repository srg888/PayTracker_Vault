# Архитектурные решения

## ADR-001. Разделение общей заявки и специализированных типов

### Статус
Принято.

### Решение
Использовать общую сущность `REQUESTS` и отдельные специализированные сущности для конкретных типов заявок (см. [[ER модель]]).

```text
REQUESTS
├── PAYMENT_REQUESTS
├── PURCHASE_REQUESTS
└── CONSULTATION_REQUESTS
```

### Причина
Общие поля хранятся централизованно. Специфичные поля не создают большое количество `NULL`-значений.

## ADR-002. Отдельный справочник типов документов

### Статус
Принято.

### Решение
Типы документов хранятся в отдельном справочнике — [[Справочник типов документов]]. Требования к документам для конкретной заявки хранятся отдельно (`REQUEST_DOC_REQUIREMENTS`, см. [[ER модель]]).

### Причина
Общие правила и индивидуальные исключения должны быть разделены (см. [[Бизнес-правила|BR-052]]).

## ADR-003. Серверная проверка прав

### Статус
Принято.

### Решение
UI не является источником истины для проверки прав. Каждый защищённый endpoint самостоятельно проверяет полномочия пользователя (см. [[Матрица прав]]).

### Причина
Пользователь может напрямую отправить HTTP-запрос, не используя интерфейс приложения.

## ADR-004. Точные денежные расчёты

### Статус
Принято.

### Решение
Использовать Decimal / NUMERIC для денежных значений (см. [[Бизнес-правила|BR-060]]).

### Причина
Бинарный `float` не обеспечивает точное представление денежных значений.

## ADR-005. Номера заявок

### Статус
Принято.

### Решение
Номер заявки не должен формироваться через `COUNT(*) + 1`. Номер должен быть основан на уникальном идентификаторе или на отдельном безопасном механизме нумерации (см. [[Бизнес-правила|BR-013]]).

### Причина
Параллельное создание заявок может привести к дублированию номеров. Удаление заявки не должно приводить к повторному использованию номера.

## ADR-006. Заявка и фактическое исполнение

### Статус
На рассмотрении.

### Вопрос
Может ли одна заявка иметь несколько [[08. Фактическое исполнение платежа|фактических исполнений]]?

### Вариант A
Одна заявка = одно исполнение.

### Вариант B
Одна заявка = одно или несколько исполнений.

### Решение
До подтверждения бизнес-сценария не усложнять модель. Если появляются частичные платежи или несколько операций по одной заявке, ввести отдельную сущность `PAYMENT_EXECUTIONS`.

## ADR-007. История назначений исполнителя

### Статус
Принято.

### Решение
Текущий исполнитель хранится в заявке. История назначений хранится отдельно (см. [[03. Назначение исполнителя|Назначение исполнителя]]).

### Причина
Необходимо поддерживать переназначение и аудит изменений.
