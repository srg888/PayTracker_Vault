# Нефункциональные требования

## Безопасность
Все операции, изменяющие данные, должны выполняться только после серверной проверки прав (см. [[Матрица прав]], [[Архитектурные решения|ADR-003]]). Секреты и ключи не должны храниться в исходном коде. Секреты production-среды должны передаваться через переменные окружения или защищённое хранилище секретов.

## Аудит
[[Аудит|Аудит]] значимых действий должен быть неизменяемым для обычных пользователей. Все записи должны иметь дату и время в единой временной зоне.

## Денежные данные
Денежные суммы должны храниться в точном десятичном формате. Расчёты не должны выполняться через бинарный `float` (см. [[Архитектурные решения|ADR-004]]).

## Файлы
Для загружаемых файлов должны быть определены: максимальный размер одного файла; максимальный общий размер; допустимые форматы; правила именования; правила хранения; правила удаления; срок хранения. Текущие лимиты для форм заявок — см. [[01. Поля заявки — Платёж|Поля заявки — Платёж]] и [[03. Поля заявки — Консультация|Поля заявки — Консультация]].

## Производительность
Для типовых операций пользовательский интерфейс должен обеспечивать приемлемое время ответа. Особое внимание необходимо уделить: спискам заявок; отчётам ([[Отчётность]]); фильтрации; поиску; загрузке истории аудита.

## Надёжность
Операции, изменяющие несколько связанных объектов, должны выполняться транзакционно. При ошибке операции система не должна оставлять частично сохранённые данные.

## Резервное копирование
Должны быть определены: периодичность резервного копирования; срок хранения копий; процедура восстановления; ответственный за восстановление.

## Масштабирование
Архитектура должна позволять: добавлять новые типы заявок; добавлять новые способы платежа; добавлять новые типы документов; добавлять новые роли; расширять отчётность. Без необходимости переписывать существующую бизнес-логику.
